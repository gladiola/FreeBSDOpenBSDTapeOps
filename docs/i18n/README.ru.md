## Скрипты

| Скрипт | Целевая ОС |
|---|---|
| `scriptedDemo.sh` | FreeBSD |
| `scriptedDemo_openbsd.sh` | OpenBSD |

Оба скрипта выполняют одну и ту же последовательность операций:

1. Просят пользователя подтвердить, что лента загружена.
2. Перематывают ленту в начало.
3. Выводят состояние ленты.
4. Показывают содержимое архивов на позициях файлов 0, 1, 2 и 3 с помощью `tar t`.
5. Переводят ленту в автономный режим.

Каждый шаг делает паузу и ждёт, пока пользователь нажмёт **Enter**, прежде чем продолжить, поэтому скрипты подходят для интерактивных демонстраций или пошаговых инструкций.

## Различия между двумя скриптами

### 1. Путь к устройству ленты

Скрипты нацелены на разные узлы устройств ленты:

- **FreeBSD** (`scriptedDemo.sh`): `/dev/nsa0`
- **OpenBSD** (`scriptedDemo_openbsd.sh`): `/dev/nrst0`

Оба являются узлами устройств без автоматической перемотки (префикс `n`), поэтому позиция ленты сохраняется между командами, а скрипты явно управляют позиционированием через `mt rewind` и `mt fsf`.

### 2. Шаг загрузки ленты

- **FreeBSD**: при запуске выполняет `mt -f /dev/nsa0 load`, чтобы механически загрузить кассету в привод перед перемоткой.
- **OpenBSD**: пропускает команду `load`, потому что `mt(1)` в OpenBSD не поддерживает подкоманду `load`. Скрипт OpenBSD предполагает, что лента уже находится в приводе, и сразу переходит к перемотке.

## Скрипты конвейера журналов OpenBSD A-to-B-to-C

Каталог `scripts/` содержит скрипты для сценария, в котором OpenBSD Computer B получает записи rsyslog от Computer A, ежедневно пакетирует их, отправляет на один из нескольких серверов Computer C, а Computer C записывает их на ленту.

| Скрипт | Назначение |
|---|---|
| `scripts/computer-b-hourly-rotate.sh` | Создаёт почасовой ротированный журнал из активного входного файла rsyslog на Computer B. |
| `scripts/computer-b-daily-archive.sh` | Упаковывает один день (`YYYYMMDD`) почасовых журналов в архив `.tar.gz` с диапазоном времени на Computer B, исключая текущий час, чтобы избежать конфликтов с активной записью. |
| `scripts/computer-b-send-archives.sh` | Отправляет неотправленные ежедневные архивы (`.tar.gz` и при необходимости `.tar.gz.enc`) с Computer B на один или несколько серверов Computer C через `scp`. |
| `scripts/computer-c-receive-archives.sh` | Проверяет входящие незашифрованные архивы и ставит незашифрованные/зашифрованные архивы в очередь для записи на ленту. |
| `scripts/computer-c-write-to-tape.sh` | Записывает поставленные в очередь незашифрованные или зашифрованные архивы на ленту, проверяет свободное место, безопасно дописывает и помечает их как записанные. |
| `scripts/computer-c-inventory-tape.sh` | Печатает таблицу содержимого ленты по файловым маркерам, чтобы операторы могли быстро находить архивы. |
| `scripts/computer-c-restore-archive-from-tape.sh` | Сканирует позиции файлов на ленте для поиска запрошенного архива, при необходимости выполняет расшифровку и сохраняет восстановленные данные в файл. |
| `scripts/test-computer-a-b-c-integration.sh` | Выполняет детерминированный локальный интеграционный тест A→B→C (включая восстановление со ленты), который не зависит от реального времени. |

Типичное расписание:

- Запускайте `computer-b-hourly-rotate.sh` каждый час (cron на B).
- Запускайте `computer-b-daily-archive.sh` один раз в день (cron на B).
- Запускайте `computer-b-send-archives.sh` после создания архива (cron на B).
- Периодически запускайте `computer-c-receive-archives.sh` на C.
- Периодически запускайте `computer-c-write-to-tape.sh` на C с правильным ленточным устройством.
- Запускайте `computer-c-inventory-tape.sh` на C, когда вам нужна таблица содержимого по маркерам.
- Запускайте `computer-c-restore-archive-from-tape.sh` на C, когда нужно восстановить конкретный архив для проверки.

Все скрипты конвейера помимо вывода в консоль также отправляют рабочие сообщения в syslog через `logger` (например, видимые через rsyslog/journaling).

### Отправка на несколько серверов с Computer B

`computer-b-send-archives.sh` поддерживает как режим одного сервера, так и режим нескольких серверов:

- Один сервер: `computer-b-send-archives.sh <archive_dir> <user@host> <remote_dir>`
- Несколько серверов: `computer-b-send-archives.sh <archive_dir> <remote_dir> <user@host> [user@host...]`

Параметры выбора сервера на стороне клиента:

- Укажите один сервер в аргументах, чтобы закрепиться за одним Computer C.
- Укажите несколько серверов, чтобы разрешить резервный переход.
- Установите `PREFERRED_SERVER=user@host`, чтобы выбрать один конкретный сервер из предоставленного списка.

Параметры обработки занятости на Computer B:

- `REMOTE_BUSY_MARKER` (по умолчанию: `.busy`): файл-маркер, проверяемый на удалённой стороне.
- `BUSY_RETRY_SECONDS` (по умолчанию: `60`): время ожидания между повторами, пока сервер занят.
- `BUSY_MAX_RETRIES` (по умолчанию: `10`): максимальное число попыток повтора для каждого сервера.

### Публикация состояния занятости со стороны Computer C

`computer-c-write-to-tape.sh` создаёт маркер занятости во время активной записи архивов на ленту и удаляет его в состоянии простоя.

- `BUSY_MARKER` (по умолчанию: `<received_dir>/.busy`)

На Computer B направьте `REMOTE_BUSY_MARKER` на расположение маркера, используемое Computer C.

### Безопасность ленты и поведение дописывания на Computer C

Перед записью каждого архива `computer-c-write-to-tape.sh` проверяет доступную ёмкость ленты/устройства и требует как минимум:

`archive_size + TAPE_SAFETY_MARGIN_BYTES`

Связанные переменные:

- `TAPE_SAFETY_MARGIN_BYTES` (по умолчанию: `10485760`)
- `TAPE_AVAILABLE_BYTES` (переопределение для известного доступного пространства)
- `ALLOW_UNKNOWN_TAPE_SPACE=1` (разрешает запись, если определить пространство не удалось)

Для реальных ленточных устройств скрипт записи перед записью перемещается к концу данных (`mt eom`/`mt eod`), поэтому несколько архивов добавляются, а не перезаписывают существующее содержимое ленты.

### Человекочитаемые отметки времени в именах файлов

- Почасовые журналы получают имена вида: `rsyslog-2026-06-01T1600.log`
- Ежедневные архивы получают имена вида: `rsyslog-2026-06-01T0000_to_2026-06-01T2300.tar.gz`

Диапазоны ежедневных архивов основываются на фактических первом и последнем почасовых файлах, включённых в архив.
Эти имена сделаны читаемыми для людей, просматривающих временные окна дат/времени событий.
Текущий час намеренно исключается из создания архива, чтобы активные записи не передавались.

### Необязательное OpenSSL шифрование для ежедневных архивов

`computer-b-daily-archive.sh` после создания tar-архива может зашифровать архивы с помощью OpenSSL:

- `OPENSSL_ENCRYPT_KEY_FILE=/path/to/keyfile` для симметричного шифрования (`openssl enc`, шифр по умолчанию `aes-256-gcm`).
- `OPENSSL_ENCRYPT_CERT_FILE=/path/to/cert.pem` для шифрования сертификатом получателя (`openssl smime`).
- `OPENSSL_ENCRYPT_CIPHER` для выбора OpenSSL шифр как в режиме файла ключа, так и сертификата (по умолчанию: `aes-256-gcm`).

Одновременно можно задать только один из этих вариантов. Зашифрованные выходные файлы используют `.tar.gz.enc`.
В целях безопасности скрипт отклоняет слабые или не-AEAD варианты шифра и требует шифры класса GCM/poly1305.

### Восстановление архива с ленты на Computer C

Используйте `computer-c-restore-archive-from-tape.sh`, чтобы найти конкретный архив, последовательно просматривая файлы на ленте с начала:

```sh
scripts/computer-c-restore-archive-from-tape.sh <tape_device> <archive_name> <output_file>
```

- Для имён архивов вида `rsyslog-<start>_to_<end>.tar.gz` (или `.tar.gz.enc`) скрипт определяет правильное совпадение, проверяя наличие пограничных почасовых файлов в восстановленном восстановленных данных.
- Если именование архивов у вас другое, задайте `TARGET_MEMBER_GLOB` как шаблон оболочки, соответствующий элементу, который обязательно должен существовать в архиве.
- Если архив зашифрован, укажите параметры дешифрования по мере необходимости:
  - `OPENSSL_DECRYPT_KEY_FILE` (симметричный режим `openssl enc`; шифр дешифрования по умолчанию: `aes-256-gcm`)
  - `OPENSSL_DECRYPT_CERT_FILE` и `OPENSSL_DECRYPT_PRIVATE_KEY_FILE` (режим дешифрования S/MIME)

Восстановленный вывод записывается как незашифрованный файл `.tar.gz`, чтобы его можно было изучить инструментами вроде `tar -tzf`.

### Инвентаризация содержимого ленты на Computer C

Используйте `computer-c-inventory-tape.sh`, чтобы вывести таблицу содержимого по маркерам:

```sh
scripts/computer-c-inventory-tape.sh <tape_device>
```

Выходные столбцы включают:

- `file_marker`: позиция файлового маркера на ленте с нуля
- `status`: `ok`, `decrypted` или `unreadable`
- `encrypted`: требовалась ли дешифровка для проверки записи (`yes`/`no`)
- `archive_hint`: предполагаемое имя в стиле архива, когда удаётся распознать границы
- `first_member` / `last_member`: первый и последний элементы tar, увиденные в этом маркере
- `member_count`: число элементы tar, найденных в этом маркере
- `bytes`: сырые байты, прочитанные на этом маркере

Это позволяет оператору определить индекс маркера, к которому нужно перейти (`mt fsf <N>`) перед операциями восстановления.

### Детерминированный интеграционный тест A/B/C

Используйте `scripts/test-computer-a-b-c-integration.sh`, чтобы проверить сквозную интеграцию компьютеров A, B и C независимо от прошедшего времени:

```sh
scripts/test-computer-a-b-c-integration.sh
```

Этот скрипт:

1. Имитирует запись журналов системой A.
2. Запускает на B ротацию и создание ежедневного архива.
3. Имитирует передачу во входящий каталог C.
4. Запускает на C приём и запись на ленту.
5. Восстанавливает архив с ленты и проверяет содержимое.

Он использует фиксированную метку дня (`TEST_DAY_STAMP`, по умолчанию `20260101`), поэтому поведение воспроизводимо и не привязано к текущей дате/времени.

### 72-часовое хранения с защитой для неподтверждённых данных

Теперь скрипты по умолчанию используют окно хранения в 72 часа:

- `computer-b-hourly-rotate.sh` удаляет старые почасовые журналы только тогда, когда существует соответствующий локальный маркер подтверждения `.taped`.
- `computer-b-send-archives.sh` удаляет старые локальные архивы только тогда, когда существуют оба маркера подтверждения: `.sent` и локальный `.taped`.
- `computer-c-write-to-tape.sh` удаляет только те старые архивы, у которых уже есть маркеры `.taped`.

В результате файлы, которые ещё не были успешно переданы и записаны на ленту, сохраняются даже если они старше `RETENTION_HOURS` (по умолчанию `72`).
На Computer B локальная очистка требует наличия локальных маркеры `.taped` (например, после этапа обратной синхронизации или ручного процесса подтверждения).
На Computer C возраст хранения измеряется по времени модификации маркера `.taped` (обычно оно устанавливается в момент успешной записи на ленту).

## Скрипти

| Скрипт | Цільова ОС |
|---|---|
| `scriptedDemo.sh` | FreeBSD |
| `scriptedDemo_openbsd.sh` | OpenBSD |

Обидва скрипти виконують однакову послідовність операцій:

1. Просять користувача підтвердити, що стрічку завантажено.
2. Перемотують стрічку на початок.
3. Виводять стан стрічки.
4. Перелічують вміст архівів у файлових позиціях 0, 1, 2 та 3 за допомогою `tar t`.
5. Переводять стрічку в автономний режим.

Кожен крок робить паузу й чекає, поки користувач натисне **Enter**, перш ніж продовжити, тому ці скрипти підходять для інтерактивних демонстрацій або керованих покрокових проходжень.

## Відмінності між двома скриптами

### 1. Шлях до пристрою стрічки

Скрипти орієнтовані на різні вузли стрічкових пристроїв:

- **FreeBSD** (`scriptedDemo.sh`): `/dev/nsa0`
- **OpenBSD** (`scriptedDemo_openbsd.sh`): `/dev/nrst0`

Обидва є вузлами пристроїв без автоматичного перемотування (префікс `n`), тому позиція стрічки зберігається між командами, а скрипти явно керують позиціонуванням через `mt rewind` і `mt fsf`.

### 2. Крок завантаження стрічки

- **FreeBSD**: під час запуску виконує `mt -f /dev/nsa0 load`, щоб механічно завантажити касету в привід перед перемотуванням.
- **OpenBSD**: пропускає команду `load`, оскільки `mt(1)` в OpenBSD не підтримує підкоманду `load`. Скрипт OpenBSD припускає, що стрічка вже знаходиться в приводі, і відразу переходить до перемотування.

## Скрипти конвеєра журналів OpenBSD A-to-B-to-C

Каталог `scripts/` містить скрипти для сценарію, у якому OpenBSD Computer B отримує записи rsyslog від Computer A, щоденно пакетно обробляє їх, надсилає на один із кількох серверів Computer C, а Computer C записує їх на стрічку.

| Скрипт | Призначення |
|---|---|
| `scripts/computer-b-hourly-rotate.sh` | Створює погодинно ротований журнал з активного вхідного файлу rsyslog на Computer B. |
| `scripts/computer-b-daily-archive.sh` | Об’єднує один день (`YYYYMMDD`) погодинних журналів у `.tar.gz`-архів із часовим діапазоном на Computer B, виключаючи поточну годину, щоб уникнути конфліктів з активним записом. |
| `scripts/computer-b-send-archives.sh` | Надсилає не надіслані щоденні архіви (`.tar.gz` і за потреби `.tar.gz.enc`) з Computer B на один або кілька серверів Computer C через `scp`. |
| `scripts/computer-c-receive-archives.sh` | Перевіряє вхідні незашифровані архіви та ставить незашифровані/зашифровані архіви в чергу для запису на стрічку. |
| `scripts/computer-c-write-to-tape.sh` | Записує поставлені в чергу незашифровані або зашифровані архіви на стрічку, перевіряє місце, безпечно дописує та позначає їх як записані. |
| `scripts/computer-c-inventory-tape.sh` | Друкує таблицю вмісту стрічки за файловими маркерами, щоб оператори могли швидко знаходити архіви. |
| `scripts/computer-c-restore-archive-from-tape.sh` | Сканує файлові позиції на стрічці для пошуку запитаного архіву, за потреби виконує розшифрування і зберігає відновлені дані у файл. |
| `scripts/test-computer-a-b-c-integration.sh` | Виконує детермінований локальний інтеграційний тест A→B→C (включно з відновленням із стрічки), що не залежить від реального часу. |

Типовий розклад:

- Запускайте `computer-b-hourly-rotate.sh` щогодини (cron на B).
- Запускайте `computer-b-daily-archive.sh` один раз на день (cron на B).
- Запускайте `computer-b-send-archives.sh` після створення архіву (cron на B).
- Періодично запускайте `computer-c-receive-archives.sh` на C.
- Періодично запускайте `computer-c-write-to-tape.sh` на C з правильним стрічковим пристроєм.
- Запускайте `computer-c-inventory-tape.sh` на C, коли потрібен таблицю вмісту за маркерами.
- Запускайте `computer-c-restore-archive-from-tape.sh` на C, коли потрібно відновити конкретний архів для перевірки.

Усі скрипти конвеєра, окрім виводу в консоль, також надсилають операційні повідомлення в syslog через `logger` (наприклад, видимі через rsyslog/journaling).

### Надсилання на кілька серверів з Computer B

`computer-b-send-archives.sh` підтримує як режим одного сервера, так і режим кількох серверів:

- Один сервер: `computer-b-send-archives.sh <archive_dir> <user@host> <remote_dir>`
- Кілька серверів: `computer-b-send-archives.sh <archive_dir> <remote_dir> <user@host> [user@host...]`

Параметри вибору сервера на боці клієнта:

- Вкажіть один сервер в аргументах, щоб закріпитися за одним Computer C.
- Вкажіть кілька серверів, щоб дозволити резервне перемикання.
- Установіть `PREFERRED_SERVER=user@host`, щоб вибрати один конкретний сервер із наданого списку.

Параметри обробки зайнятості на Computer B:

- `REMOTE_BUSY_MARKER` (типово: `.busy`): файл-маркер, який перевіряється на віддаленому боці.
- `BUSY_RETRY_SECONDS` (типово: `60`): час очікування між повторами, поки сервер зайнятий.
- `BUSY_MAX_RETRIES` (типово: `10`): максимальна кількість повторних спроб для кожного сервера.

### Публікація стану зайнятості з Computer C

`computer-c-write-to-tape.sh` створює маркер зайнятості під час активного запису архівів на стрічку та видаляє його, коли система неактивна.

- `BUSY_MARKER` (типово: `<received_dir>/.busy`)

На Computer B спрямуйте `REMOTE_BUSY_MARKER` на розташування маркера, яке використовує Computer C.

### Безпека стрічки та поведінка дописування на Computer C

Перед записом кожного архіву `computer-c-write-to-tape.sh` перевіряє доступну місткість стрічки/пристрою і вимагає щонайменше:

`archive_size + TAPE_SAFETY_MARGIN_BYTES`

Пов’язані змінні:

- `TAPE_SAFETY_MARGIN_BYTES` (типово: `10485760`)
- `TAPE_AVAILABLE_BYTES` (перевизначення для відомого доступного простору)
- `ALLOW_UNKNOWN_TAPE_SPACE=1` (дозволяє запис, якщо простір неможливо визначити)

Для реальних стрічкових пристроїв скрипт запису перед записом переходить до кінця даних (`mt eom`/`mt eod`), тому кілька архівів додаються, а не перезаписують наявний вміст стрічки.

### Людинозрозумілі позначки часу в іменах файлів

- Погодинні журнали мають назви на кшталт: `rsyslog-2026-06-01T1600.log`
- Щоденні архіви мають назви на кшталт: `rsyslog-2026-06-01T0000_to_2026-06-01T2300.tar.gz`

Діапазони щоденних архівів ґрунтуються на фактичних першому та останньому погодинних файлах, включених до архіву.
Ці назви призначені бути зрозумілими людям, які переглядають вікна дат/часу подій.
Поточну годину навмисно виключено зі створення архіву, щоб активні записи не передавалися.

### Необов’язкове OpenSSL шифрування для щоденних архівів

`computer-b-daily-archive.sh` може шифрувати архіви за допомогою OpenSSL після створення tar-архіву:

- `OPENSSL_ENCRYPT_KEY_FILE=/path/to/keyfile` для симетричного шифрування (`openssl enc`, шифр за замовчуванням `aes-256-gcm`).
- `OPENSSL_ENCRYPT_CERT_FILE=/path/to/cert.pem` для шифрування сертифікатом отримувача (`openssl smime`).
- `OPENSSL_ENCRYPT_CIPHER` для вибору OpenSSL шифр як у режимі файла ключа, так і сертифіката (типово: `aes-256-gcm`).

Одночасно можна встановити лише один із цих варіантів. Зашифровані вихідні файли використовують `.tar.gz.enc`.
З міркувань безпеки скрипт відхиляє слабкі або не-AEAD варіанти шифру і вимагає шифри класу GCM/poly1305.

### Відновлення архіву зі стрічки на Computer C

Використовуйте `computer-c-restore-archive-from-tape.sh`, щоб знайти конкретний архів, послідовно переглядаючи файли стрічки від початку:

```sh
scripts/computer-c-restore-archive-from-tape.sh <tape_device> <archive_name> <output_file>
```

- Для імен архівів на кшталт `rsyslog-<start>_to_<end>.tar.gz` (або `.tar.gz.enc`) скрипт визначає правильний збіг, перевіряючи наявність граничних погодинних файлів у відновленому відновлених даних.
- Якщо ваші назви архівів інші, установіть `TARGET_MEMBER_GLOB` як шаблон оболонки, що відповідає елементу, який обов’язково має існувати в архіві.
- Якщо архів зашифрований, надайте параметри дешифрування за потреби:
  - `OPENSSL_DECRYPT_KEY_FILE` (симетричний режим `openssl enc`; шифр дешифрування за замовчуванням: `aes-256-gcm`)
  - `OPENSSL_DECRYPT_CERT_FILE` і `OPENSSL_DECRYPT_PRIVATE_KEY_FILE` (режим дешифрування S/MIME)

Відновлений результат записується як незашифрований файл `.tar.gz`, щоб його можна було перевірити інструментами на кшталт `tar -tzf`.

### Інвентаризація вмісту стрічки на Computer C

Використовуйте `computer-c-inventory-tape.sh`, щоб надрукувати таблицю вмісту за маркерами:

```sh
scripts/computer-c-inventory-tape.sh <tape_device>
```

Стовпці виводу містять:

- `file_marker`: нульова позиція файлового маркера на стрічці
- `status`: `ok`, `decrypted` або `unreadable`
- `encrypted`: чи потрібне було дешифрування для перевірки запису (`yes`/`no`)
- `archive_hint`: припустима назва у стилі архіву, коли можна розпізнати межі
- `first_member` / `last_member`: перший і останній елементи tar, побачені в цьому маркері
- `member_count`: кількість елементи tar, знайдених у цьому маркері
- `bytes`: сирі байти, прочитані на цьому маркері

Це дає оператору змогу визначити індекс маркера, до якого треба перейти (`mt fsf <N>`) перед операціями відновлення.

### Детермінований інтеграційний тест A/B/C

Використовуйте `scripts/test-computer-a-b-c-integration.sh`, щоб перевірити наскрізну інтеграцію комп’ютерів A, B і C незалежно від минулого часу:

```sh
scripts/test-computer-a-b-c-integration.sh
```

Цей скрипт:

1. Імітує запис журналів системою A.
2. Запускає на B ротацію та створення щоденного архіву.
3. Імітує передавання до вхідного каталогу C.
4. Запускає на C приймання та запис на стрічку.
5. Відновлює архів зі стрічки та перевіряє вміст.

Він використовує фіксований штамп дня (`TEST_DAY_STAMP`, типово `20260101`), щоб поведінка була відтворюваною і не залежала від поточної дати/часу.

### 72-годинне зберігання із захистом для непідтверджених даних

Тепер скрипти типово використовують 72-годинне вікно зберігання:

- `computer-b-hourly-rotate.sh` видаляє старі погодинні журнали лише тоді, коли існує відповідний локальний маркер підтвердження `.taped`.
- `computer-b-send-archives.sh` видаляє старі локальні архіви лише тоді, коли існують обидва маркери підтвердження: `.sent` і локальний `.taped`.
- `computer-c-write-to-tape.sh` видаляє лише ті старі архіви, які вже мають маркери `.taped`.

У результаті файли, які ще не були успішно передані та записані на стрічку, зберігаються навіть тоді, коли вони старші за `RETENTION_HOURS` (типово `72`).
На Computer B локальне очищення вимагає локальних маркери `.taped` (наприклад, після кроку зворотної синхронізації або ручного процесу підтвердження).
На Computer C вік зберігання вимірюється за часом модифікації маркера `.taped` (зазвичай його встановлюють у момент успішного запису на стрічку).
